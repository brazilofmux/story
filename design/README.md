# Design

Architectural sketches and design notes for the story-telling engine. Each sketch is numbered and dated; revisions do not overwrite predecessors. Old sketches are left in place as a record of how the design changed, so the reasoning over time is preserved.

## Convention

- Files named `{topic}-sketch-NN.md` where NN is sequential per topic. A *topic* is a coherent design area — `substrate`, `architecture`, `descriptions`, and so on. Each topic has its own sequence.
- Every sketch has a **Status** (*draft, active*, *superseded*, *abandoned*) and a **Date** at the top.
- **Active sketches are self-contained within their topic.** A reader should be able to understand the current state of *that topic* from the active sketch alone, without chasing a chain of deltas. When corrections pile up, consolidate into a new sketch rather than let the active starting point depend on superseded context.
- **Cross-topic sketches are expected.** Some sketches (notably architecture-level ones) apply across topics. They do not supersede the sketches they constrain; they add a frame that future topic sketches are expected to satisfy. Reading a topic sketch alone gives the reader that topic's current state; reading the cross-topic sketches additionally gives them the frame those topic sketches sit inside.
- When a sketch is superseded, its status is updated and a link to the successor is added — but the file stays.
- Sketches are not specifications. They are the design thinking in progress. Claims and commitments may be wrong. Open questions are first-class.

- [Dramatic dialect — sketch 01](dramatic-sketch-01.md) — first upper-dialect sketch under architecture-sketch-02's stack. Sandwich question (whether anything sits between Dramatica and substrate) explicitly left open.

## Upcoming sketches

Planned work. Each is a sketch-to-write; filenames are provisional. Dependency order is approximate; sketches are advanced when a forcing function argues for it (e.g., the descriptions surface shipped write-only, which advanced reader-model; Jocasta's anagnorisis had a clean failure mode, which advanced identity-and-realization; the `project_reader` weakening had a clean forcing function in both encodings, which advanced focalization; the Oedipus parricide/incest encoding exercised the compound-predicate pinch, which advanced the inference model).

1. **Second Dramatic-dialect encoding** (provisional) — dramatic-sketch-01's Oedipus example is one data point. Pressuring the dialect with a structurally dissimilar story (Macbeth for clearer Antagonist shape; *The Brothers Karamazov* for multi-Argument polyphony; *The Remains of the Day* for Main-Character-only throughline) will surface dialect gaps Oedipus can't. Likely the next forcing function for dramatic-sketch-02.
2. **Structural dialect — sketch 01** (provisional) — parallel upper dialect to Dramatic. Acts, beats, plot points, pacing markers — the McKee / Aristotelian vocabulary. Deliberately separate from Dramatic (M-sketch-01 excludes act structure and pacing on purpose): authors who want structural scaffolding without argumentative apparatus, or vice versa, should be able to take one without the other. Written standalone on its own terms, same discipline as dramatic-sketch-01.
3. **Lowering-record shape — sketch 01** (deferred) — exact fields of the `Lowering` record from architecture-sketch-02. Deliberately held until at least two upper dialects exist and have been independently pressured, so the lowering shape can be designed against concrete source records rather than speculatively.
4. **Inference model — sketch 02** (provisional) — follow-ons to inference-sketch-01 as forcing functions emerge. Likely candidates: typed inequality / NAF for `sibling`-family rules; rule-anchored descriptions (`AnchorRef.kind == "rule"`); rule attention and rule staleness; refining N9 under culturally-loaded compound predicates.
5. **View vs. sjuzhet visibility** (provisional, banked from the Oedipus probe walk) — the `ReaderView` exposes substrate state (τ_s, τ_a) but not the sjuzhet schedule (τ_d). Descriptions that reference disclosure-schedule coordinates ("the reader has held both since τ_d=0") make claims the view cannot verify. Resolution shape unclear: either the view grows sjuzhet visibility, or descriptions are disciplined to stay inside view coordinates. Touches architecture-sketch-02's OQ4 (cross-dialect query semantics) and is called out as a prerequisite in that sketch's *What happens next* (item 4) — the cross-boundary verifier's primitives depend on whether τ_d is queryable. See `D_view_cannot_see_τ_d` in the Oedipus encoding.

The reader-model probe now exercises the view against live Claude Opus 4.6 (`reader_model_client.py`, `demo_reader_model.py` for Rashomon, `demo_reader_model_oedipus.py` for Oedipus), ingests structured reviews, answer proposals, and edit proposals as substrate-native records, and surfaces a proposal queue the author can walk interactively (`proposal_walker.py`, driven by `--walk`). The walker routes accept decisions through `ingest_review`, `accept_answer_proposal`, and `accept_edit_proposal`; declines through `decline_proposal`. Edit acceptance supersedes the source description (marks it `DescStatus.SUPERSEDED`, hides from default views) and commits a new record at a fresh τ_a, with `supersedes` / `superseded_by` metadata pointers preserving the chain — closing the bounded case of descriptions-sketch-01 OQ1 (description versioning) and reader-model-sketch-01 OQ6 (description-edit proposals). Persistence of the queue across sessions is still deferred per descriptions-sketch-01 OQ3.

Inference-model-sketch-01 was seeded by a probe walk against the Oedipus encoding: the probe's answer to `D_parricide_incest_authored_not_derived` supplied the derivable-boundary criterion (N9). The next unit of prototype work is implementing the `Rule` record, `holds_derived` / `derive_all` query surface, and proof-carrying derivation described in that sketch, then retiring the Oedipus encoding's author-asserted `parricide` / `incest` compound predicates and re-probing to see whether the probe reaches for richer rule families (sibling, uncle) that would pressure OQ1 or OQ3.
